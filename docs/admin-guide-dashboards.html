Dual Data Sources

Why two data sources?

Prometheus provides real-time metrics with 15-day retention — ideal for current status, trend charts, and instant queries. Postgres stores structured health-check logs with 90-day retention — enabling historical audit trails and detailed error messages that Prometheus' label-based model cannot efficiently store.

3.1 HTTP Probes (Blackbox)

HTTP http_2xx_or_401 (primary module)

How it works: Prometheus instructs the Blackbox Exporter to perform an HTTP GET to the target URL. The exporter connects, sends the request, and evaluates the response status code against the module's valid_status_codes list.

  • Success criteria: HTTP status 200 or 401 → probe_success = 1
  • TLS handling: Certificate verification is skipped (insecure_skip_verify: true) to monitor endpoints with self-signed certificates
  • User-Agent: Monitor-Monitor/2.0-Grafana
  • Timeout: 30 seconds per probe

Why 401 is accepted: Many monitored endpoints (IMS, CGMS, Keycloak) require authentication. A 401 Unauthorized confirms the service is running and responding — the check verifies availability, not authentication.

Output metrics:

MetricTypeMeaning
probe_successGauge1 = target responded with valid status, 0 = failure
probe_duration_secondsGaugeTotal probe round-trip time in seconds
probe_ssl_earliest_cert_expiryGaugeUnix timestamp of the soonest SSL certificate expiry date
probe_http_status_codeGaugeHTTP status code returned by the target

3.2 TCP Connectivity Checks (Blackbox)

TCP tcp_connect module

How it works: The exporter opens a TCP socket to host:port. A successful three-way handshake (SYN → SYN-ACK → ACK) means the port is open and accepting connections.

  • Timeout: 10 seconds
  • Used for: Postgres database ports (3306, NodePorts 30399/30306)
  • Does NOT authenticate — only verifies the port is reachable

3.3 ICMP Ping (Blackbox)

ICMP icmp_ping module

How it works: Sends an ICMP Echo Request packet. A reply confirms the host is powered on and network-reachable.

  • Timeout: 5 seconds
  • Scrape interval: 2 minutes (more frequent than HTTP/TCP)
  • Preferred IP: IPv4
  • Used for: Kubernetes master/worker nodes, lab infrastructure hosts

3.4 LDAP Directory Checks (Monitor Exporter)

LDAP Custom Python Exporter

How it works: The exporter uses the ldap3 Python library to attempt an anonymous bind to the LDAP server.

Important: Bind rejection = UP

A bind rejection (e.g., "anonymous bind disallowed") means the LDAP server is running and actively refusing the request. This is counted as UP. Only connection-level failures (timeout, connection refused, DNS failure) are DOWN.

3.5 Keycloak Health Checks (Monitor Exporter)

KEYCLOAK Custom Python Exporter

How it works: Sends an HTTP GET to the public, unauthenticated endpoint /auth/realms/master.

Success criteria (all must be true):

  1. HTTP response status is 200 OK
  2. Response body is valid JSON
  3. JSON contains a "realm" field matching the expected realm name (typically master)

Version detection priority:

  1. X-Keycloak-Version response header
  2. Server header (regex match)
  3. Fallback: /.well-known/openid-configuration endpoint

Output metrics: monitor_keycloak_up, monitor_keycloak_response_time_ms, monitor_keycloak_realm_valid

3.6 Database Connectivity Checks (Monitor Exporter)

DATABASE Custom Python Exporter

How it works: Opens a raw TCP socket to the Postgres/MySQL port. The MySQL protocol sends a greeting packet (handshake) before any authentication is required. The exporter parses this packet to:

  • Confirm the database service is running (monitor_database_up)
  • Extract the server version string from the greeting (bytes 5+)
  • Measure response time
No credentials required

The MySQL/Postgres protocol sends the server version in the initial handshake packet before authentication. This means the exporter can detect the database version and confirm availability without any database credentials.

Dashboard refresh vs. scrape interval

The dashboards auto-refresh every 1 minute, but HTTP/TCP data only updates every 5 minutes. This means refreshing the dashboard between scrapes will show the same data. This is expected behavior, not a bug.

Health Status variable — how it works

The $health_status variable uses a PromQL operator injection technique. The variable values (>= 0, == 1, == 0) are appended directly to PromQL expressions using the :raw format modifier. For example, selecting "DOWN" transforms the query from probe_success{...} to (probe_success{...}) == 0, which returns only series with value 0 (DOWN systems).

For the Postgres SQL query, the variable value is safely compared using :sqlstring format (which prevents SQL injection) and mapped to the UP/DOWN string values stored in Postgres.

Data Links (Clickable Filtering)

The "Healthy" and "Down" stat values are clickable. Clicking them sets the health_status variable to filter the tables below:

  • Clicking "Healthy" → navigates to ?...&var-health_status=%3D%3D%201 (URL-encoded == 1)
  • Clicking "Down" → navigates to ?...&var-health_status=%3D%3D%200 (URL-encoded == 0)

The links use ${__all_variables} to preserve all current dashboard variable selections (time range, system_group, log_rows).

Scope note

These counts only reflect job="blackbox_http" (HTTP probes). TCP, LDAP, Keycloak, and Database systems are not included in the stat counts but are included in the filtered table below. This means clicking "Healthy 24" may show more than 24 rows in the table.

TABLE All Monitored Systems Panel ID: 20

The most comprehensive panel. Combines four different metric types into a single table using Prometheus instant queries and Grafana transformations.

Query A — Status (determines UP/DOWN)

(probe_success{job=~"blackbox_http|blackbox_tcp", system_group=~"$system_group"}
  or monitor_ldap_up{system_group=~"$system_group"}
  or monitor_keycloak_up{system_group=~"$system_group"}
  or monitor_database_up{system_group=~"$system_group"}
) ${health_status:raw}

The ${health_status:raw} suffix appends the comparison operator. When "All" is selected, this becomes ... >= 0 (matches both 0 and 1). When "DOWN" is selected, it becomes ... == 0 (only DOWN systems).

  or monitor_ldap_response_time_ms{...} / 1000
  or monitor_keycloak_response_time_ms{...} / 1000

Note: Monitor Exporter metrics are in milliseconds and divided by 1000 to match Blackbox's seconds unit.

TABLE Recent Health Check Logs Panel ID: 40

Queries Postgres for historical health check entries logged by the Custom Python Exporter. Shows LDAP, Keycloak, and Database check results.

SQL Query

SELECT
  hch.check_timestamp AS `Time`,
  ms.display_name     AS `System`,
  ms.system_group     AS `Group`,
  ms.system_type      AS `Type`,
  hch.status          AS `Status`,
  hch.response_time_ms AS `Response`,
  hch.error_message   AS `Error`
FROM health_check_history hch
JOIN monitored_system ms ON hch.system_id = ms.id
WHERE $__timeFilter(hch.check_timestamp)
  AND ms.system_group IN (${system_group:sqlstring})
  AND (
    ${health_status:sqlstring} = '>= 0'
    OR (${health_status:sqlstring} = '== 1' AND hch.status = 'UP')
    OR (${health_status:sqlstring} = '== 0' AND hch.status = 'DOWN')
  )
ORDER BY hch.check_timestamp DESC
LIMIT $log_rows

SQL Injection Protection

Safe variable interpolation

The :sqlstring format modifier wraps the variable value in quotes and escapes any embedded quotes. The query then compares the escaped string against hardcoded literals ('>= 0', '== 1', '== 0'). Any unexpected value simply fails to match any branch, returning no rows — a safe failure mode.

Health Status Filtering Logic

Variable ValueSQL Behavior
>= 0 (All)First OR branch matches → no status filter applied
== 1 (UP)Second branch matches → only hch.status = 'UP' rows
== 0 (DOWN)Third branch matches → only hch.status = 'DOWN' rows
Any other valueNo branch matches → no rows returned (safe failure)

9. Data Correctness & Validation

This section explains how to verify that the dashboards display accurate data and identifies known edge cases.

9.1 Verifying Prometheus Metrics

  1. Check target status: Open http://localhost:9090/targets. All targets should show UP with a recent "Last Scrape" timestamp. Targets showing DOWN indicate the exporter is unreachable (not the target system itself).
  2. Query raw metrics: In Prometheus UI (http://localhost:9090/graph), execute probe_success to see all current values. Each result should have system_id, display_name, system_group labels.
  3. Compare counts: Execute count(probe_success{job="blackbox_http"} == 1) and verify it matches the "Healthy" number in Dashboard Panel 1.
  4. Check Monitor Exporter: curl http://localhost:9116/metrics should return all monitor_* metrics. Verify monitor_ldap_up, monitor_keycloak_up, and monitor_database_up values match expected state.

9.2 Verifying Postgres Logs

-- Count recent entries (should be ~(targets × checks_per_hour) per hour)
SELECT COUNT(*) FROM health_check_history
WHERE check_timestamp > NOW() - INTERVAL 1 HOUR;

-- Verify all systems are logging
SELECT ms.system_id, ms.display_name, MAX(hch.check_timestamp) AS last_check
FROM monitored_system ms
LEFT JOIN health_check_history hch ON ms.id = hch.system_id
GROUP BY ms.system_id, ms.display_name
ORDER BY last_check;

9.3 Known Edge Cases

Panel 1 count scope mismatch

The "Healthy" and "Down" counts in Panel 1 only count job="blackbox_http" (HTTP probes). The "All Monitored Systems" table (Panel 20) also includes TCP, LDAP, Keycloak, and Database systems. This means clicking "Healthy 24" may show a different number of rows in the filtered table.

or vector(0) fallback

Queries like count(probe_success == 0) or vector(0) use or vector(0) to return 0 when no systems are down (since count() of an empty set returns no data, not 0). If this fallback is missing, the stat panel would show "No data" instead of "0".

SSL expiry calculation accuracy

The formula (probe_ssl_earliest_cert_expiry - time()) / 86400 computes days until expiry using the current evaluation time. Because SSL metrics are scraped hourly, the displayed days can be up to 1 hour stale. This is acceptable for certificate monitoring.

9.4 Security: Variable Interpolation Safety

VariableFormatInjection RiskMitigation
$system_group Default (PromQL regex) Low — values come from Prometheus label query Prometheus escapes regex special characters
$system_group :sqlstring (SQL) None — values are quoted and escaped :sqlstring prevents SQL injection
$health_status :raw (PromQL) Medium — raw injection into PromQL Values are from a fixed custom list (not user-editable). URL manipulation with invalid values returns empty results.
$health_status :sqlstring (SQL) None — compared as escaped string literal Value is wrapped in quotes; unrecognized values fail safely (no rows returned)
$log_rows Default (numeric) Low — fixed numeric options Non-numeric values cause SQL error (no injection vector)
